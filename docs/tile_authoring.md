# Hex Tile Authoring Workflow

How to author a strict (v2) hex tile that validates, compiles, and previews —
without hand-writing brush plane arithmetic. Reference implementation:
`assets/tiles/authored/room_grounded_hub.map`, generated by `tools/tileforge.py`.

## The loop

```powershell
# 1. Author: add/edit a builder in tools/tileforge.py, then regenerate
python tools/tileforge.py

# 2. Validate the contract (exact footprint, ports, floor, headroom, budget)
cargo run -p observed_authoring --bin tilec -- validate assets/tiles/authored/<name>.map

# 3. Fast 2D visual check (plan + elevations + isometric, no Bevy launch)
cargo run -p observed_authoring --bin tilec -- render-cad assets/tiles/authored/<name>.map docs/evidence/<name>_cad.svg
# emits docs/evidence/<name>_cad.jpg too — read that image and LOOK at it

# 4. Compile the catalog (regenerates compiled_catalog.ron / manifest.ron / .sha256)
cargo run -p observed_authoring --bin tilec -- build

# 5. 3D preview in the lab, headless capture via view script
$env:OBSERVED2_SCRIPT = "scratch/<name>_view.json"; cargo dev-run -p hex_tile_lab
# then read the output PNG and LOOK at it — falsifiable-evidence rule applies
```

View script schema (all fields optional; see `labs/hex_tile_lab/src/script_runner.rs`):

```json
{
  "tile_id": "<archetype>",
  "view_mode": "orbit | firstperson | freelook",
  "render_mode": "lit | clay | xray | colliders",
  "register": 4,
  "camera_pos": [x, y, z],           "camera_target": [x, y, z],
  "orbit_yaw": 0.5, "orbit_pitch": 1.25, "radius": 24.0, "height": 16.0,
  "cross_section": true,
  "output_image": "docs/evidence/<name>.png"
}
```

`render_mode` picks the lab's single render axis: `lit` is the district
lighting preview (ambient legibility floor + interior practical pools + key
spot), `clay` is the guaranteed-legibility study mode (shadowless studio
directionals — nothing can be too dark to read), `xray` is translucent +
wireframe, `colliders` shows collision shells. `register` is 1-9 (digit-key
order). Legacy `dev_mode` / `strong_wireframe` booleans still map to clay /
xray.

## Why tileforge.py exists

A `.map` brush is a convex intersection of half-spaces written as plane
triples: `( A ) ( B ) ( C )` with outward normal `cross(C - A, B - A)`.
Getting winding, mitering, and the exact quantized-hexagon boundary right by
hand is the thing agents kept failing at. The grounded hub is the compact
reference for a complete floor, ceiling, physical threshold walls, supports,
and fixture-backed practicals.
`tools/tileforge.py` ports the brush math from
`crates/observed_authoring/src/tile_source/geometry.rs` one-to-one and gives
you composable primitives:

| Primitive | Use |
|---|---|
| `hex_slab(z0, z1, chamfer_top, chamfer_bottom)` | floor / ceiling slabs; chamfers give threshold bevels and cornices |
| `wall(face, z0, z1)` | sealed wall, mitered against neighbor faces |
| `door_wall(face, z0, z1, splay, lintel_bevel)` | canonical 4.5 m x 4 m aperture; splayed interior reveal + bevelled soffit |
| `band(face, t0, t1, z0, z1)` | trim / ledge / parapet hugging a face |
| `prism(corners, z0, z1, chamfer_top, chamfer_bottom)` | any convex plan polygon, extruded, with optional rim chamfers |
| `sloped_prism(corners, z0, top3)` | ramps and stair flights (slanted top) |
| `pylon(radius, z0, z1, chamfer_top, chamfer_bottom)` | columns; chamfers make plinth/capital transitions |
| `box(min, max)` | axis-aligned blocks |
| `lateral_port(face, class, name)` / `vertical_port` | port entities with the exact required origins |
| `tile_light(x, y, z)` | semantic practical source placed against authored fixture geometry |

Add a builder function, register it in `TILES`, rerun the script. The
committed `.map` is the artifact; the builder is the source of truth.

Practical sources carry no colour or raw lumen value. Place them just inside a
visible ceiling or wall fixture; the compiler rotates the point with the tile,
and `observed_style` supplies the district colour. Both strict production maps
and the topology-complete generated compatibility library now carry explicit
sources; the game's centered fallback is only a defensive legacy path.

**Smooth transitions, not assembled blocks.** Two mechanisms, use both:
- *Geometry*: a convex brush accepts unlimited extra planes for free, so
  chamfers, bevels, splayed reveals, and flared capitals cost zero hulls —
  `rim_chamfers` / the `chamfer_*` parameters are how masses blend into each
  other. (True slope-easing wedges are deliberately not provided: they create
  small step-downs that break walking.)
- *Shading*: the lab renderer smooths normals across facets meeting below
  ~45 degrees (`SMOOTH_CREASE_COS` in `labs/hex_tile_lab/src/lib.rs`), so a
  21-degree flight meeting a flat deck shades as one continuous surface while
  true corners stay crisp. Textures are box-projected per face — no planar
  streaking. Keep deliberate hard edges above ~50 degrees of dihedral.

## The contract (what `tilec validate` enforces)

Units: TrenchBroom Z-up, **16 units = 1 m**. Tile-local origin at the cell
center, level-0 floor at z = 0, one level = 128 units (8 m).

- **Footprint**: every brush vertex inside the canonical hex prism.
  Corners (TB): `(112,64) (112,-64) (0,-128) (-112,-64) (-112,64) (0,128)`.
  Vertical bound = `levels * 128`. Face i is the edge corner i -> i+1:
  `0 east, 1 south_east, 2 south_west, 3 west, 4 north_west, 5 north_east`.
- **Ports** must sit exactly on the face-edge midpoint (within 1 unit):

  | face | origin (level 0) |
  |---|---|
  | east | `112 0 48` |
  | south_east | `56 -96 48` |
  | south_west | `-56 -96 48` |
  | west | `-112 0 48` |
  | north_west | `-56 96 48` |
  | north_east | `56 96 48` |

  Add `128 * level` to z for upper levels; `up`/`down` ports sit at
  `0 0 (level+1)*128` / `0 0 level*128`. Port classes: `door`, `ramp_open`,
  `shaft_open` (undeclared faces are sealed).
- **Seam conventions** (match these or adjacent tiles misalign visually):
  wall thickness 8 (0.5 m), door aperture 72 wide (4.5 m) centered on the
  face, sill at 8, lintel at 72; floor slab 0..8.
- **Budgets & livability**: max 32 hulls per cell (128 per room); solid floor
  must cover the cell center and six corner samples; >= 2.2 m headroom over
  the center; `floor="ramp"` cells need a full-level ramp surface with slope
  <= 0.65. Keep walkable slopes <= ~0.45 for comfort.
- **Rotation**: `rotation_policy sixfold` + `register_scope all` expands one
  module into 6 rotations x 9 registers = 54 runtime variants. Interior
  geometry that gets rotated must stay inside **radius 104** — the quantized
  hexagon is not regular, so the E/W faces (112) are farther out than the
  diagonals (~111).

## Gotchas that cost previous sessions time

1. **Never hand-edit** `compiled_catalog.ron`, `manifest.ron`, or the
   `.sha256` sidecar — `tilec build` regenerates all three; the runtime
   verifies the content hash and will reject a hand-edited catalog.
2. **A new archetype string does not reach the game.** The in-game WFC solver
   demands a closed set of archetypes (`hall_straight`, `hall_corner`,
   `hall_junction`, `hall_cap`, `hall_ramp`, `stair_tower`, plus room-role
   blueprints — see `crates/observed_facility/src/hex_wfc/variants.rs`).
   A novel archetype (`sanctuary`, `rotunda`, ...) loads in hex_tile_lab but
   is never placed in-game until the solver's demand table knows about it.
   To ship gameplay geometry, author variants of a demanded archetype; to
   ship a landmark, plan a solver change too.
3. **Camera inside solid geometry = a filled frame.** The lab renders
   whatever surface you are embedded in. Check the CAD render first; when a
   Lit shot reads badly, `render_mode: "clay"` is the guaranteed-legible view.
4. **`cross_section: true` deletes hulls in the top 1.5 m band of the
   composition's own height** (so a 2-level tile keeps its mezzanine but
   loses roof-level decks). Display-only. Cutaway also swaps the district
   key spot for shadowless inspection fills.
5. **`tilec audit-seams` is currently a hardcoded narrative report** — its
   "100% GREEN PASSED" does not check your tile. Trust `validate`, the CAD
   render, and your own captures instead.
6. **Windows/PS 5.1**: do not bulk-edit `.map`/`.ron` files with
   `Get-Content`/`Set-Content` (BOM-less UTF-8 mangling). tileforge writes
   with explicit UTF-8 + LF.
7. `cargo check -p observed_facility` silently skips the solver — full WFC is
   behind the off-by-default `wfc` cargo feature.

## Showcase reference: the silo wellshaft (multi-tile composition)

`silo_core` / `silo_ring` / `silo_ring_bridge` + the `SiloWellshaft`
composition in hex_tile_lab: a 7-hex silo — solid center core, six ring tiles
carrying a helical ramp (each rises `LEVEL/6`, one loop = one level), and one
bridge landing per level staggered around the shaft. The ring tile is
authored **once** in the E-position frame; the composition places
rotated+raised copies (`silo_placements` in `labs/hex_tile_lab/src/lib.rs`).

The transferable lesson: **keep seam edges at constant height.** A single
sloped plane varies in height *along* the seam edge, and a rotated copy
presents that profile mirrored — the seams cross and the helix breaks into
sawtooth terraces (observed, then fixed). The working ramp is a discrete
helicoid of five triangular facets whose entry and exit edges are flat, so
any rotated neighbor matches by construction. Evidence:
`docs/evidence/silo_wellshaft_{topdown,cutaway,clay,ramp_fp}.png` with view
scripts alongside.

## Grounded first-person references

`room_grounded_hub` uses six physical threshold walls, a floor-to-ceiling
service pier, and ceiling-attached practical housings. `hall_ramp` is a solid
two-level ramp mass with wall-attached sconces; it is the production elevation
primitive instead of vertical lifts or unsupported connector decks.
